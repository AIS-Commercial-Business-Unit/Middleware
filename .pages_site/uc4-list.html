UC4 — GetAppraisalList

Scatter-gather fan-out to @Work (SQL) and DEIPDE07 (IBM MQ mainframe) in parallel. Results merged, deduplicated, and returned — partial on timeout.

Try it: GET /api/appraisal/list?policyNumber=POL-BOTH-001  ·  Flow Tracer: localhost:3000/ops/{correlationId}

Architecture

How It Works

Fan-Out

DocumentListSaga receives GetAppraisalDocumentListCommand and immediately sends two messages: GetAtWorkDocumentListCommand and a fire-and-forget to the DEIPDE07 IBM MQ stub via ArtemisAdapter. Both branches run in parallel — neither waits for the other.

@Work Branch

AtWorkDocumentListHandler queries SQL Server, publishes AtWorkDocumentListCompletedEvent back to the saga. Typically completes in <100ms.

DEIPDE07 Branch

The IBM MQ stub sends one or more reply messages, each with a position indicator (N of M). ArtemisListReplyListener (always-on BackgroundService) receives them and publishes MainframeDocumentListCompletedEvent.

Merge & Timeout

Once both branches complete, the saga merges and deduplicates by documentkey. If DEIPDE07 does not respond within the timeout, the saga completes with partialResult=true using only @Work data.

Policy NumberExpected ResultWhat to Show
POL-BOTH-001 4 records · partialResult=false Both branches complete. @Work returns 2 RiskID records, DEIPDE07 returns 2 numeric records. Position indicators: "1 of 2", "2 of 2".
POL-SQL-001 3 records · partialResult=false DEIPDE07 returns 0 messages. @Work-only result. No error raised.
POL-MQ-001 0 records · partialResult=false Mainframe-only result. All records have numeric document keys.
POL-TIMEOUT-001 2 records · partialResult=true DEIPDE07 times out. @Work results returned immediately. Timeout logged with elapsed time.

BizTalk comparison: In BizTalk the fan-out is inside orchestration shapes with no queryable state. A timeout either crashes the orchestration or is swallowed silently. Here the timeout is handled gracefully and every step is a structured log entry.
