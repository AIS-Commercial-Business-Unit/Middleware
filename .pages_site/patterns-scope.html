Pattern Scope — What Was Reviewed

All four BizTalk estates were reviewed. This page explains what was found, what was prioritized, and why.

The BizTalk documentation covered four business areas: SCI (Small Commercial Insurance), PRS (Personal Risk Services), ECOS (Accenture Claims), and Sanctions (EMEA/APAC/LATAM). Every application in each estate was reviewed. Patterns were classified by architectural risk — not by business priority.

Two use cases were selected for implementation. These were chosen because together they cover the patterns that represent the highest migration risk across all four estates. They are not the most common pattern — API-to-API request/reply is far more common. They are the hardest pattern, and proving the hard patterns work demonstrates that the straightforward ones will too.

Why These Two Use Cases

UC-1 — Policy Issuance demonstrates:

  • Durable long-running workflow state (Saga survives process failure)
  • Compliance hold and resume — workflow suspended for up to 72 hours, automatically resumed on external event
  • Decentralized message routing — each downstream system self-selects via subscription filter, no central routing table
  • Multi-system parallel fan-out with aggregation
  • Dead-letter recovery — failed messages routable and replayable from the portal
  • End-to-end distributed tracing through all systems

These capabilities apply to SCI Issuance, SCI Quote Proposal, PRS Appraisal UpdateStatus, and Sanctions Clearance — all share the same architectural pattern: a workflow that must survive interruption and resume from external state.

UC-3 — Automated Renewal Batch demonstrates:

  • File-triggered batch processing — Blob Storage arrival triggers chunked processing
  • Per-record failure isolation — one bad record does not stop the batch
  • Horizontal scale-out under load — Container Apps replicas increase automatically under queue pressure
  • Timer-based scheduling — replaces AutoSys and BizTalk scheduled receive locations
  • Batch Saga state tracking — batch progress recorded across potentially 50,000+ individual records

These capabilities apply to SCI AutomatedRenewal, ECOS Outbound Transaction Workflow, PRS PLUS Notes batch, and PRS @Work 15-minute polling cycle — all share the same architectural pattern: scheduled polling, per-record isolation, and exactly-once delivery tracking.

What Was Not Built and Why

Pattern TypePresent InNot Built Because
Simple API-to-API (WCF/REST request-reply) All four estates — majority of interfaces The lowest architectural risk. Any integration framework handles synchronous request-reply. There is nothing to prove here beyond what UC-1 already demonstrates.
Document generation (PRT1X3, xPression, FileNet) SCI QuoteProposal External vendor dependency. The platform pattern (async Saga triggering an external document service and storing the result) is demonstrated in UC-1. The vendor-specific endpoint is a configuration detail.
SFTP file exchange PRS Claim Care (Tymetrix, Sterling), PRS PMSI Medicare Azure Logic Apps Standard provides a built-in SFTP connector. This is a solved problem. Not architecturally interesting.
BRE (Business Rules Engine) SCI (DecidePolicy), Sanctions (Clearance BRE) The DecidePolicy routing table is replaced entirely by Service Bus subscription filters in the AIS+ architecture — the BRE is eliminated, not migrated. The Sanctions BRE becomes a rule service on AKS. Both approaches are documented in Pattern Detail.
Full PRS Appraisal lifecycle (UpdateStatusSaga) PRS RiskID The most complex flow in PRS. The Saga pattern it uses is identical to UC-1. The additional complexity is in the business rules (UW determination branching) and multi-database state, not in the integration platform. A production implementation would follow the same Saga structure proven in UC-1.
