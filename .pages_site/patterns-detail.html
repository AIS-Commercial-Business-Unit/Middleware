All patterns from all four estates. Criticality, POC coverage, and AIS+ implementation approach for each.

Each estate is listed below. Expand to see the full pattern table. Criticality reflects architectural migration risk, not business priority. High = the pattern has no trivial equivalent in the target platform and requires deliberate design. Medium = well-understood; implementation is straightforward but must be done. Low = direct mapping; essentially a configuration change.

Multi-system parallel fan-out with aggregation (QuoteProposal) High QuoteProposalSaga fans out to DuckCreek CI/ELPL/Auto, ForeFront, Insurity simultaneously. Aggregates all quote responses before triggering document generation. Documented QuoteProposalSaga: publishes QuoteDataRequested to Service Bus topic. Each PAS gateway responds with QuoteDataReceived. Saga counts responses against expected count (or 10-min timeout). When complete, publishes QuoteReadyForDocumentation to Platform.DocumentGeneration.
Dual-channel notification (Kafka + MQ simultaneously) Medium QuoteProposal sends EVN12X1 event to both Kafka HTTP gateway and MQSC queue. Two different consumer systems receive the same event. Documented Platform.Notification domain dispatches to multiple channels from a single PublishNotificationIntent command. Kafka consumers receive via Service Bus Kafka surface. Legacy MQ consumers receive via IBM MQ transport. Channel disabled per-consumer when migration is complete — configuration change, no code change.
Dead-letter and correlated retry (BTMS) Medium Failures route to BTMS via MQSC. BTMS creates work item for manual review. Resume by replaying original request using AccountServiceRequestNumber. POC Built NServiceBus dead-letter queue. ServicePulse UI shows failed messages with full context. Operator replays from portal. IssuanceSaga resumes from last confirmed step. .NET → Java →
IBM MQ inbound (MQSC receive port) Medium RcvQueue MQSC receive port on EPL39X1. Primary inbound transport for IssuePolicyRequest. Documented NServiceBus IBM MQ transport (Particular Software, officially supported). Receives from MQ queue, publishes to Service Bus. Coexistence: MQ remains active; new consumers subscribe to Service Bus. Legacy senders require no change.
Simple API-to-API (WCF endpoints) Low WCF-WebHttp and WCF-BasicHttp adapters calling ERM7X1, CRM19X1, CRM40X1, xPression, PRT1X3. Not Required HTTP client in .NET calls the same endpoints. APIM provides authentication, rate limiting, and retry policy. No adapter layer needed.

PRS — Personal Risk Services  49 apps · RiskID: 19 · Claim Care: 30 · 34 vendor integrations
Pattern Criticality BizTalk Implementation POC AIS+ Implementation Approach
Appraisal lifecycle Saga (UpdateStatus from RiskID) High Inbound MQSC from RiskID (StatusCode 6 = UW determination, StatusCode 15 = complete). StatusCode 6: parallel PLUW calls, creates suspense items, updates @Work/PLAPR/PLUW databases. Most complex flow in PRS. Documented UpdateStatusSaga in Appraisal domain. Starts on RiskIDStatusUpdateReceived event. Branches on StatusCode. StatusCode 6: sends GetProducerCrossReference to Customer domain, then parallel PLUW calls (Task.WhenAll equivalent). Updates three databases via adapter microservices. Same Saga pattern as UC-1 — proven architecture, different business rules.
Scatter-gather (GetAppraisalList) High Fan-out to @Work SQL SP and DEIPDE07 simultaneously using MQSC correlation sets (MQMD_CorrelId / MQMD_MsgId). Merge both responses. Documented Platform.Integration dispatches GetAppraisalData to @Work SQL adapter and DEIPDE07 adapter simultaneously. Service Bus session-based correlation replaces MQMD correlation. Aggregates both responses; returns partial result with PartialResult=true if one backend times out. Identical pattern to NServiceBus scatter-gather.
Content-based routing (GetAppraisalDocument) High Routes on DocumentKey format: RiskID-format → RiskID system; ACE-format → RSVP; other → DEIPDE08. Decision in BizTalk orchestration code. Documented C# switch statement on DocumentKey format in the Appraisal domain message handler. Routes to RiskID adapter, RSVP adapter, or DEIPDE08 adapter in Platform.Integration. Readable, testable code — not a BizTalk decision shape.
DB2-triggered integration (AtWorkImport, PlusNotes) High BizTalk DB2 adapter polls DB2 database for change events. Triggers @Work import and PLUS Notes sync. Documented Azure Function with DB2 polling using Net.IBM.Data.Db2 NuGet (IBM-published, supports .NET 6–10, Linux containers). Publishes events to Service Bus on change detection. Db2Connect license required (~$1K–$36K depending on edition).
15-minute polling batch (@Work update) Medium BizTalk orchestration polls PLAPR DB every 15 minutes. Deduplication by timestamp. Updates @Work, marks processed. POC Built Timer-triggered Azure Function (15-min interval). Queries PLAPR via Azure SQL adapter. Per-record deduplication preserved. Same batch isolation pattern as UC-3. .NET →
Real-time address verification (Claim Care) Medium WCF-Service endpoint. ACC calls synchronously on address entry. User waits for response. Target: sub-500ms. Documented APIM endpoint in Claims domain. Calls Zip+4 validation service via HTTP. Graceful degradation: if service unavailable, returns input address with IsVerified=false — user workflow is never blocked. Load tested at 50 concurrent requests.
SFTP bulk invoice processing (Tymetrix, Sterling) Medium BizTalk SFTP receive port. Processes daily invoice files. Sends reverse status feeds back. Not Required Azure Logic Apps Standard built-in SFTP connector triggers on file arrival. Publishes to Service Bus. Claims domain processes and sends reverse feed. Direct connector — no custom code.
Safelite FNOL (synchronous, returns ClaimNumber) Medium WCF-Service synchronous call from HSG. Creates claim in ACC. Returns ClaimNumber and adjuster in same response. Zero async. Documented APIM endpoint in Claims domain. Synchronous: HTTP POST → ACC integration → return ClaimNumber. No Saga needed. APIM handles retry, timeout, and authentication. Standard request-reply.
OFAC payment screening (Claim Care) Medium Two trigger paths: WCF-Service (direct) and WCF-SQL time-based polling. Routes to CLink compliance service. Documented Claims domain delegates to Platform.Compliance (same domain used by SCI Issuance). OFACClaimsCheck aggregate tracks the check. ComplianceClearanceSaga holds payment pending result. Billing domain releases payment on ComplianceCleared.
Simple WCF API-to-API (majority of Claim Care) Low 22 of 30 Claim Care apps are synchronous WCF-Service endpoints. Standard request-reply with no state. Not Required APIM endpoint + HTTP client. No integration framework needed. Claims domain exposes REST API; ACC calls it directly.
ECOS — Accenture Claims Platform  4 apps · 10–12K transactions/hour · SQL + MSMQ + DB2 triggers
Pattern Criticality BizTalk Implementation POC AIS+ Implementation Approach
High-volume SQL polling batch (TX_ACTIVITY) High BizTalk WCF-SQL polls TX_ACTIVITY for N/R/H records. Updates status to I (In Progress). Returns activity ID list. Runs continuously. 10–12K tx/hour. POC Built Azure Function with timer trigger. Queries TX_ACTIVITY via Azure SQL using same stored procedure logic. Updates status to I atomically. Publishes activity IDs to Service Bus. Same per-record isolation as UC-3. .NET →
Activity/staging DB pattern (TX_ACTIVITY → TX_MESSAGE_OUTBOUND) High Three-table staging chain: TX_ACTIVITY (RunDB) → TX_MESSAGE_OUTBOUND (Integration DB) → TX_PROCESS_OUTBOUND (tracking). CycleID-based batch coordination. BizTalk writes to Integration DB after enrichment. Documented Service Bus outbox pattern replaces the three-table chain. Message handler reads TX_ACTIVITY, calls PullDataService to enrich, publishes to Service Bus (writes to TX_MESSAGE_OUTBOUND equivalent atomically). TX_PROCESS_OUTBOUND replaced by Service Bus delivery tracking per subscription. Azure SQL retains the tables for audit if required.
Factory-pattern data enrichment (PullDataService) High PullDataServiceProvider uses Factory pattern. GetInstance(transactionType) creates concrete IPullDataServiceiCGBC per tx type (RegisterClaim, UpdateClaim, PostPayment, etc.). WCF call to ussbyintvt2040.aceins.com:8071/PullDataService.svc. Documented C# factory or strategy pattern in the ECOS integration domain. Each transaction type has a concrete ITransactionProcessor implementation. HTTP client calls PullDataService with activity ID. Response dispatched to the appropriate handler by C_TXTYPE. Same pattern as PRS content-based router — proven approach.
MSMQ polling batch trigger Medium BizTalk MSMQ adapter polls for batch trigger messages. Used to coordinate between BizTalk orchestrations. Documented Service Bus queue replaces MSMQ. NServiceBus MSMQ transport provides a bridge during coexistence period — existing MSMQ senders require no change. When MSMQ is decommissioned, the bridge is removed.
SSIS/ETL export pipeline (AutoSys + SSIS) Medium AutoSys schedules SSIS Master Package per interface. Master → ET child (extract/transform from Integration DB) → L child (load to file/FTP/DB). Updates TX_PROCESS_OUTBOUND on completion. Documented Azure Data Factory pipeline replaces SSIS. Timer trigger (Azure Function or ADF schedule) replaces AutoSys. ADF reads TX_MESSAGE_OUTBOUND, applies transformation activity, writes to Blob/FTP/downstream DB. Updates processing status on completion. Existing SSIS packages can be lifted into ADF SSIS Integration Runtime as a migration phase if full rewrite is not immediate.
DB2 integration triggers Medium DB2 adapter in BizTalk ECOS estate. Specific integration types triggered by DB2 change events. Documented Same approach as PRS DB2: Azure Function with Net.IBM.Data.Db2 NuGet. Polls DB2 for change events, publishes to Service Bus. Db2Connect license required.
FTP outbound delivery Low SSIS L child package FTPs output files to external endpoints. Not Required Azure Logic Apps Standard built-in FTP connector sends files after ADF processing. Or direct FTP client in Azure Function. Either is straightforward.
Sanctions — EMEA / APAC / LATAM  5 apps · 3 regional deployments · ~18K tx/day/region · 24×7 SLA
Pattern Criticality BizTalk Implementation POC AIS+ Implementation Approach
Compliance hold Saga (ClearanceProcessor) High BizTalk orchestration suspended on RSK3X3 PendingReview. Inbound RSK4X1 MQSC message (from CAHSEMEA) delivers clearance. BRE routes to Cadence or UWP. MaxWait: 72 hours. Documented ComplianceClearanceSaga in Platform.Compliance — same Saga proven in UC-1 for SCI. Holds CorrelationId on PendingReview. IBM MQ polling via NServiceBus MQ transport receives RSK4X1 clearance. Saga publishes ComplianceCleared or auto-blocks after 72-hour timeout. BRE routing replaced by conditional dispatch in C# based on clearance attributes.
Synchronous compliance search (ScreenProcessor) High WCF-Service endpoint. WorkView calls sync. CAHSEMEA calls async with callback. Transforms to RSK3X3 SearchAndUpdateForClaim. Calls CLink. Returns StatusCode to caller. ~18K/day/region. Documented APIM endpoint in Platform.Compliance. Synchronous path: HTTP → CLink call → return result. Async path: HTTP → publish to Service Bus → callback when complete. Same endpoint, two response modes based on caller preference. APIM handles authentication, rate limiting, and regional routing.
Multi-subject payment screening (TraxScreenProcessor) High TRAX payment → canonical map produces 4 subjects (Individual payee, Org payee, Bank BIC, Bank name) → single CLink call with all 4 as EconomicSanctionRisk entities. ASPAC only. Documented Platform.Compliance accepts TraxScreenRequest. Transformer maps TRAX payment to canonical with 4 subjects. Single CLink call. Response mapped back to TRAX format. Azure Function or AKS container — same pattern as ScreenProcessor.
IBM MQ inbound polling (ClearanceProcessor) High BizTalk MQSC receive port polls IBM MQ queue for RSK4X1 inbound clearance notifications from CAHSEMEA. Documented NServiceBus IBM MQ transport polls the same MQ queue. On message arrival, deserializes RSK4X1 payload, publishes SanctionsClearanceReceived event to Service Bus. ComplianceClearanceSaga correlates by SourceSystemTransactionId. During migration: both BizTalk and NServiceBus can poll the same queue using different consumer group IDs.
BRE-driven routing (Clearance BRE) Medium CHUBB.Sanctions.Clearance.BRE determines target system (Cadence or UWP) based on clearance notification attributes. Documented BRE rules translated to C# conditional logic in the clearance message handler. Rules are few and stable — a code-based implementation is more transparent and testable than an external rules engine for this use case. If rules complexity grows, Azure Logic Apps Decision Tables or a custom rule service can replace the C# conditions.
Multi-region deployment (EMEA / APAC / LATAM) Medium Three separate BizTalk deployments, one per region. Regional configuration via SSO (ESRFileSearchCode, ESRResolutionTeamName, BranchCode). Documented Azure Service Bus geo-redundancy + regional AKS node pools. Regional configuration stored in Azure App Configuration with environment-specific labels (EMEA, APAC, LATAM). Same container image deployed to each region; configuration differs. IaC (Bicep) provisions all three regions from a single template.
Message transformation (canonical maps) Medium BizTalk maps in CHUBB.Sanctions.Compliance, CHUBB.Sanctions.Compliance.Trax, CHUBB.Sanctions.Clearance.Transforms. NtificationRq_To_CadenceRq, TraxSanctions_Canonical_To_CLink maps. Documented C# model classes with explicit mapping logic. Canonical domain model as the pivot. Source-to-canonical and canonical-to-target mappers are plain classes — unit testable, version-controllable, readable. No BizTalk XSLT or BTM map required.
SSO runtime configuration (regional) Low BizTalk SSO stores ESRFileSearchCode, ESRResolutionTeamName, BranchCode per application. Read at orchestration startup. Not Required Azure App Configuration with environment labels. All regional values stored as key-value pairs. Applications read at startup via .NET configuration provider. No code change required.
